# Skill 最佳实践

本文档总结 Skill 开发的最佳实践，帮助创建高质量的新媒体创作 Skills。

## SKILL.md 编写最佳实践

### 1. 简洁明了

**原则**：SKILL.md 应该简洁有力，控制在 200 行以内。

```markdown
# ✅ 好的做法
## 适用场景
- 撰写公众号原创长文
- 将素材扩写为完整文章
- 针对特定主题生成文章

# ❌ 不好的做法
## 适用场景
本 Skill 适用于以下场景：首先，当你需要撰写公众号原创长文时，
可以使用本 Skill 来帮助你快速生成高质量的文章内容。其次，
当你有一些素材或大纲，需要将其扩写为完整的文章时...
```

### 2. 触发词明确

**原则**：在 description 中列出用户可能使用的触发词。

```yaml
# ✅ 好的做法
description: 公众号长文写作助手。当用户说"写公众号文章"、"写一篇推文"、"帮我写篇公众号"时激活。

# ❌ 不好的做法
description: 这是一个帮助用户写作的工具。
```

### 3. 示例丰富

**原则**：提供基础和进阶用法示例，让用户快速上手。

```markdown
# ✅ 好的做法
## 使用方法

**基础用法：**
```
帮我写一篇公众号文章，主题是"远程办公的利与弊"
```

**进阶用法（指定类型和风格）：**
```
写一篇公众号干货文，主题是"新手如何学Python"，风格轻松幽默，目标读者是大学生
```

# ❌ 不好的做法
## 使用方法
输入主题即可生成文章。
```

### 4. 输出格式清晰

**原则**：明确说明输出的结构和内容。

```markdown
# ✅ 好的做法
## 输出格式

输出包含以下部分：

1. **标题**（含 1-2 个备选标题）
2. **摘要**（用于公众号摘要栏，35 字以内）
3. **正文**（含小标题分段、重点加粗）
4. **引导语**（文末引导关注/转发）

# ❌ 不好的做法
## 输出格式
输出一篇完整的文章。
```

### 5. 平台规则完整

**原则**：包含相关平台的限制和规范，避免生成违规内容。

```markdown
# ✅ 好的做法
## 平台规则

- 公众号文章无硬性字数上限，但建议 1500-5000 字为最佳阅读区间
- 标题上限 64 个字符
- 摘要上限 120 个字符
- 避免诱导分享的话术（如"不转不是中国人"）
- 避免绝对化用语（如"最好"、"第一"、"100%"）

# ❌ 不好的做法
## 平台规则
遵守公众号规则。
```

## 目录组织最佳实践

### 1. 合理使用 scripts/

**适合放入 scripts/ 的内容**：

- 内容模板（templates.md）
- AI 提示词（prompts.md）
- 检查清单（checklist.md）
- 公式库（formulas.md）

```
scripts/
├── templates.md      # 文章/笔记/脚本模板
├── prompts.md        # AI 写作提示词
├── checklist.md      # 发布前检查清单
└── formulas.md       # 标题/开头公式库
```

### 2. 合理使用 references/

**适合放入 references/ 的内容**：

- 爆款研究数据
- 平台规则详解
- 行业最佳实践
- 学习资源推荐

```
references/
├── README.md         # 主参考文档
├── case-studies.md   # 案例研究（可选）
└── resources.md      # 资源链接（可选）
```

### 3. 文件命名规范

| 文件类型 | 命名规范 | 示例 |
|----------|----------|------|
| 主文件 | SKILL.md | SKILL.md |
| 模板文件 | {类型}.md | templates.md |
| 参考文件 | README.md 或 {主题}.md | README.md |

## 内容质量最佳实践

### 1. 爆款公式要实用

```markdown
# ✅ 好的做法
### 数字 + 结果型
- 「{emoji}{数字}款{品类}，{效果描述}！」
- 示例：「✨5款平价面霜，用完皮肤嫩得能掐出水！」

# ❌ 不好的做法
### 公式1
使用数字和结果来写标题。
```

### 2. 最佳实践要具体

```markdown
# ✅ 好的做法
## 最佳实践

- **开头黄金 3 行**：前 3 行决定读者是否继续阅读，用痛点/悬念/数据开头
- **小标题分段**：每 300-500 字设置一个小标题，降低阅读压力
- **重点加粗**：核心观点和关键数据加粗处理

# ❌ 不好的做法
## 最佳实践

- 写好开头
- 分段清晰
- 突出重点
```

### 3. 数据要有来源

```markdown
# ✅ 好的做法
### 10w+ 文章共性

基于对 1000+ 篇 10w+ 阅读量文章的分析：

| 特征 | 占比 | 说明 |
|------|------|------|
| 数字型标题 | 67% | 标题中包含具体数字 |
| 痛点开头 | 72% | 前3行直击读者痛点 |

# ❌ 不好的做法
### 爆款特征
大部分爆款文章都有数字标题和痛点开头。
```

## 上下文效率最佳实践

### 1. 避免重复

**原则**：不要在 SKILL.md 中重复通用信息。

```markdown
# ✅ 好的做法
## 平台规则
- 标题上限 20 个字符
- 正文上限 1000 字
（其他通用规则参考 references/）

# ❌ 不好的做法
## 平台规则
小红书是一个生活方式分享平台，用户主要是年轻女性...
（大段介绍平台背景）
```

### 2. 分层组织

**原则**：核心内容放 SKILL.md，详细内容放 scripts/ 和 references/。

```
SKILL.md          → 核心功能、基础用法（必读）
scripts/          → 模板、公式（按需使用）
references/       → 研究数据、详细规则（深入学习）
```

### 3. 控制长度

| 文件 | 建议长度 | 说明 |
|------|----------|------|
| SKILL.md | < 200 行 | 核心内容，必须精简 |
| templates.md | < 300 行 | 模板集合，可以较长 |
| references/README.md | < 500 行 | 参考资料，可以详细 |

## 测试验证最佳实践

### 1. 多场景测试

```markdown
## 测试场景

### 基础场景
- 输入：写一篇公众号文章，主题是AI
- 预期：生成完整文章

### 进阶场景
- 输入：写一篇干货文，主题是Python，风格幽默
- 预期：生成指定类型和风格的文章

### 边界场景
- 输入：写文章（无主题）
- 预期：询问主题或给出建议
```

### 2. 质量评估

```markdown
## 质量评估维度

| 维度 | 权重 | 评估标准 |
|------|------|----------|
| 准确性 | 30% | 内容是否准确、无错误 |
| 实用性 | 30% | 输出是否可直接使用 |
| 规范性 | 20% | 是否符合平台规则 |
| 创意性 | 20% | 内容是否有吸引力 |
```

```markdown

4. 重新测试验证

```

## 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 输出太长 | 未限制长度 | 在 SKILL.md 中明确字数要求 |
| 格式不一致 | 格式说明不清 | 提供详细的输出格式模板 |
| 内容不相关 | 触发词太宽泛 | 精确定义触发词和适用场景 |
| 违反规则 | 规则说明缺失 | 补充完整的平台规则 |
| 质量不稳定 | 指令不够明确 | 添加更多约束和示例 |
